# 나는 왜 글을 못 올릴까? 
뇌과학으로 풀어본 개발자의 글쓰기 탐구

지나가는 생각들이나 경험들을 기록하는것은 의미있다고 생각한다.

흥미로운 기술을 접할 때마다 "이거 정리해서 올려야지" 하는 생각이 든다. 프로젝트에서 삽질하다가 해결책을 찾으면 "이건 꼭 공유해야 해" 하는 마음도 생긴다. 머릿속엔 쓰고 싶은 주제가 넘쳐난다. LQIP를 구현하면서 느낀 시행착오들, FSD 아키텍쳐 활용기, Astro.js를 사용하면서 느낀 점, 그 외에도 개발하면서 마주친 크고 작은 깨달음들.

하지만 실제로는 글을 많이 올리지 못했다. 글을 쓰기 시작하면 이상하게도 여러가지 늪에 빠진다. 늪에서 허우적거리다가 나오고보면 일상의 다른 중요한 일들을 하느라 결국 그 글은 올리지 못하고 지나가버리게 된다.

게다가 사실 검증이 발목을 잡는다. "이게 정확히 맞나?" 싶어서 공식 문서를 뒤지고, 레퍼런스를 찾고, 실험 코드를 짜보다 보면 하루가 간다. 처음 느꼈던 "오! 이거 재밌는데?" 하는 임팩트는 점점 희미해지고, 글을 완성할 동기도 함께 사라진다.

결국 메모 폴더엔 시작만 해놓은 초안들이 쌓여간다. 5줄짜리도 있고, 여러가지 관점의 의견과 자료를 방대하게 모아놓은 폴더로 이루어진 것도 있다. 점점 많아지는 올리지 못한 글들을 볼 때마다 아쉬움이 든다. "결국에는 못 올리는 글이 너무 많다?"


왜 못 올릴까? 
어떻게 좀더 나아질까?

글을 올리는게 왜 중요한지 다시 되새기는 것만으로는 부족하다.
그런 되새김은 이전에도 계속 해왔다.
그렇다면 어떤 행동을 통해 개선할 수 있을까? 에 초점을 맞춰본다.

왜 글을 올려야 하는지 다시 되새기는 것만으로는 부족하다. 그래서 이번엔 다르게 접근해보기로 했다. 

최근 뇌과학적인 분석을 통해 행동을 개선하는 전략을 많이 접하면서 인상깊게 본 영상이 있었는데, 비슷한 방식으로 접근해보면 좋겠다고 생각했다.

의지나 노력의 문제가 아니라, 뇌과학적 관점에서 원인을 찾아보? 그리고 실제로 행동할 수 있는 구체적인 해결책을 찾아보자.


## 글을 못 올리는 뇌과학적인 이유

### 1. 인지 부하 과부하 
> 뇌의 작업 메모리 고려하지

전전두엽(Prefrontal Cortex, PFC)은 뇌에서 "실행 관리자" 역할을 한다. 계획을 세우고, 결정을 내리고, 충동을 억제한다. 문제는 이 친구의 처리 용량이 생각보다 작다는 것이다.

글을 쓸 때 내 뇌가 동시에 처리하려는 것들:

- 글의 전체 구조 설계하기
- 지금 쓰는 문장 다듬기
- 사실 관계 검증하기
- 다른 주제로 튀지 않게 억제하기
- "이 글이 별로면 어쩌지" 불안감 관리하기
- 완벽한 표현 찾기

이건 마치 RAM이 4GB인 컴퓨터에서 Chrome 탭 30개를 동시에 열어놓은 것과 같다. 당연히 프리징이 온다. 전전두엽이 과부하되면 실행 기능이 마비되고, 뇌는 가장 편한 선택을 한다. **회피.**

"나중에 하지 뭐" 하고 창을 닫는다.

### 2. 도파민 시간은 증발한다

글감을 처음 떠올릴 때면 미묘하게 짜릿한 느낌이 있다.
"오, 이거 글로 올리면 좋겠다!"  바로 도파민이 분비되는 순간이다.

문제는 시간이 지나면 이 도파민 예측 보상이 급격히 감소한다는 것이다. 뇌과학에서는 이를 **시간 할인(Temporal Discounting)**이라고 부른다. 지금 당장의 보상은 크게 느껴지지만, 미래의 보상은 가치가 할인된다.

특히 내가 "잘 모르는 주제"에 끌리는 이유도 여기 있다. 불확실성은 도파민 기대치를 높인다. "알아내면 엄청 재밌을 거야!" 하지만 실제로 알아내려면 시간이 오래 걸린다. 그 사이 도파민은 증발하고, 완성 확률은 0에 수렴한다.

- 처음: "이 주제 완전 재밌다! 당장 써야지!" (도파민 폭발)
- 3일 후: "음... 뭐 그렇게까지 재밌나?" (도파민 감소)
- 1주일 후: "이게 뭐였지?" (동기 소멸)


### 3. 목표 희석 효과 - 주제가 계속 바뀐다. 

"React useEffect 정리"를 쓰려고 시작했는데, 어느새 "React 생명주기 전체", "클래스형 vs 함수형 컴포넌트 비교", "상태 관리 패턴", "성능 최적화"를 모두 다루려고 하고 있다.

이건 **목표 희석 효과(Goal Dilution)**와 **DMN(Default Mode Network, DMN)** 과활성 때문이다.

[DMN](https://en.wikipedia.org/wiki/Default_mode_network)은 "아무것도 안 할 때" 활성화되는 뇌 네트워크다. 멍때릴 때나 샤워할 때 활성화되면서 자유 연상을 한다. 창의적 아이디어를 만들어내는 긍정적 역할도 하지만, 글쓰기 중에 과도하게 활성화되면 문제가 된다.

A 주제를 쓰다가 → B가 떠오르고 → C도 관련 있고 → D도 써야 할 것 같고... 인지적 억제 조절이 실패하면서 초점이 흐려진다. 결국 "완벽한 종합 가이드"를 쓰려다가 한 줄도 완성하지 못한다.

### 4. 사실 검증의 늪. 완벽주의의 함정

"이 내용이 정확한가?" 하는 의심이 시작되면 끝이 없다.

React 공식 문서를 읽고, GitHub 이슈를 뒤지고, Chat GPT에 질문하고, 실제 코드를 짜보고, 다른 블로그 글들과 비교하고... 이렇게 한 문장 쓰는 데 1시간이 걸린다.

완벽주의는 편도체의 과활성과 관련이 있다. "틀린 정보를 올리면 어떡하지?" "사람들이 비판하면 어떡하지?" 하는 불안이 검증 강박으로 이어진다. (편도체는 불안을 관리하는 뇌 부위로 알려져 있다.)

문제는 기술 분야에서 "100% 정확한 설명"은 거의 불가능하다는 것이다. 버전이 바뀌고, 구현이 달라지고, 컨텍스트가 다르면 답도 달라진다. 완벽을 추구하다 보면 영원히 발행 버튼을 누를 수 없다.

_[삽화 제안: 끝없이 펼쳐진 문서와 탭들, 사실 검증의 미로에 갇힌 모습]_

## 그럼 어떻게 해야 할까? 실천 가능한 4가지 전략

뇌과학적 원인을 알았으니, 이제 뇌의 작동 방식을 역이용해보자. 
실천 가능한 4가지 전략이다.
### 1. 글쓰기를 5단계로 쪼개기

한 번에 모든 걸 하려니까 뇌가 터진다. 그렇다면 단계를 완전히 분리하면 된다.

글쓰기를 5단계로 쪼개면 **인지 부하 분산**이 이루어지면서 실행이 훨씬 가벼워진다.

**5단계 글쓰기 프로세스:**

**1단계: 브레인 덤프** (5-10분)

- 생각나는 모든 것을 비구조화된 bullet으로 쏟아내기
- 문법 무시, 순서 무시
- 단 하나의 규칙: 10분 안에 끝내기

**2단계: 주제 고정** (3분)

- 핵심을 **한 문장**으로 정의
- 예: "React useEffect cleanup 함수가 언제 실행되는지 실험해본 결과"
- 이 문장이 글의 북극성이 된다

**3단계: 뼈대 잡기** (5분)

- 3-5개 소제목만 나열
- 내용은 쓰지 않음
- 구조만

**4단계: 한 섹션씩 작성** (15분/섹션)

- 다른 섹션 생각 금지
- 타이머 15분 맞추고 하나만 집중
- 완벽하게 쓰지 않아도 됨

**5단계: 사실 검증** (마지막에만)

- 불확실한 부분에만 "[검증 필요]" 표시
- 발행 직전에 한 번에 검증

**핵심은 절대 한 번에 2개 이상의 단계를 하지 않는 것이다.** 월요일엔 1-2단계만, 화요일엔 3단계만. 이렇게 나누면 전전두엽 부하가 관리 가능한 수준으로 떨어진다.

_[플로우차트 제안: 5단계를 순차적으로 보여주는 다이어그램, 각 단계의 소요 시간 표시]_

### 2. 도파민 활용: 임팩트 느끼는 즉시 기록

도파민이 분비되는 그 순간, 그 짜릿한 임팩트를 느끼는 순간이 골든 타임이다. 3일 후가 아니라 **바로 그 순간** 기록해야 한다.

**3분 날것 캡처 시스템:**

```
[스마트폰 메모 or 음성 녹음]
- 지금 느낀 임팩트: "오! [...]"
- 핵심 아이디어 3줄
- 나중에 찾을 키워드

시간 제한: 3분
완벽함 금지
```

이건 글이 아니다. 단지 **임팩트를 보존**하는 것이다. 나중에 이 메모를 보면 그때의 흥미가 일부나마 되살아난다.

그리고 중요한 건, 이 초벌 메모를 "나중에 다듬을 것"으로 쌓아두지 말고, **바로 다음 단계로 넘어가는 트리거**로 사용하는 것이다.

### 3. 주제 확산 차단: 주차장 시스템

글 쓰다가 다른 주제가 떠오르는 건 DMN의 자연스러운 작동이기 때문에 억지로 막을 수 없다. 대신 **외부로** 분리하면 된다. 

**"주차장(Parking Lot)" 파일 운영:**

```markdown
# 주차장.md

- TypeScript 제네릭 고급 패턴 정리
- Next.js 13 App Router 마이그레이션 경험
- 브라우저 렌더링 파이프라인 깊게 파기
- 함수형 프로그래밍 실전 적용기
```

글 쓰다가 "아, 이것도 써야지!" 하는 생각이 들면:

1. 주차장 파일에 한 줄 추가
2. **즉시 현재 글로 돌아옴**

이렇게 하면 뇌는 "기록했으니 됐다"며 안심한다. **자이가르니크 효과(미완성 과제에 대한 긴장)** 가 외부화되면서 작업 기억이 해방된다.

_[이미지 제안: 주차장 파일 예시 스크린샷, 간단한 메모 형태]_

### 4. 완벽주의 무력화: 범위를 명확히 하기

완벽한 글은 없다. 대신 **명확한 범위의 글**은 있다.

**글 시작 템플릿:**

```markdown
# [주제]

⚠️ 이 글의 범위
- 이 글은 [구체적 상황]에서 제가 경험한 것을 기록한 것입니다
- [전체 주제]를 다루지 않습니다
- [버전/환경] 기준입니다

[본문...]

## 여전히 궁금한 것
- [...]
- [...]
```

이렇게 범위를 먼저 밝히면:

- 독자 기대치가 조정된다
- "불완전함"을 솔직히 말하면 오히려 신뢰도가 올라간다
- 심리적 부담이 확 줄어든다

그리고 "경험 기록"으로 프레이밍하면 검증 부담이 거의 사라진다. 내가 겪은 건 사실이니까. 틀릴 수가 없다.

_[예시 제안: 실제 블로그 글의 범위 명시 부분 스크린샷 또는 목업]_

## 진짜 현실적인 고민들

전략을 세웠다고 모든 게 해결되는 건 아니다. 실제로 적용하려니 또 다른 고민이 생긴다.

### 고민 1: 날것만 계속 쌓이면 어떡하지?

가장 현실적인 걱정이다. 3분 캡처를 열심히 했는데, 정작 완성은 안 하면 또 다른 미완성 목록이 생기는 것 아닌가?

**"1 in, 1 out" 규칙:**

```
초벌 메모 최대 5개까지만 허용
새로운 메모 만들려면:
→ 기존 것 하나 발행 or 삭제
```

강제로 우선순위를 결정하게 만든다. "나중에"의 무한 확장을 막는다.

**"부패 날짜" 시스템:**

```
파일명: 2025-10-30-react-concurrent.md
부패일: 2025-11-06 (7일 후)

→ 7일 후에도 안 쓴 메모는 자동 알림
→ 발행 or 삭제 결정
```

7일 후에도 안 쓴 글은 사실상 안 쓴다. 시스템이 정한 규칙이니 삭제해도 죄책감이 덜하다.

**가장 현실적인 해결책: 애초에 "발행 기준"을 낮추기**

초벌 메모를 쌓을 필요가 없다. 5-10분 만에 쓸 수 있는 형태로 바로 발행하면 된다.

```markdown
# TIL: Next.js Image 컴포넌트 lazy loading 이슈

오늘 발견한 것: priority 속성 없으면 LCP 지표에 안 잡힘
해결: 메인 이미지에 priority={true} 추가
결과: LCP 2.1s → 1.4s

[코드 스니펫]

참고: https://...
```

이 정도면 10분이다. 완벽하지 않아도 된다. "오늘 배운 것"이니까.

_[링크 제안: TIL 스타일로 유명한 개발 블로그 사례 - jbee.io, overreacted.io 등]_

### 고민 2: 내가 정확히 모른다는 걸 공개하는 게 두렵다

이게 진짜 핵심이다. 블로그는 공개 플랫폼이고, "전문성"에 대한 압박이 있다. "잘 모른다"고 말하면 초보로 보이지 않을까?

**뇌과학적 배경:** 편도체가 사회적 평가 위협을 감지하면 회피 반응을 일으킨다. 문제는 뇌가 부정적 피드백을 실제 확률보다 5-10배 과대평가한다는 것이다(**부정적 편향, Negativity Bias**).

**실제 데이터:**

- 가장 인기 있는 개발 블로그들: "배운 것 기록" 스타일
- 댓글 톤: 대부분 긍정적 또는 정보 추가
- 비판적 댓글: 극소수, 무시 가능

**현실적 해결책 3가지:**

**A. 주제를 "경험"으로 프레이밍**

```markdown
❌ "React 18 완벽 가이드"
✅ "프로젝트 업데이트하면서 마주친 React 18 이슈 3가지"

→ 경험은 틀릴 수 없다
→ 검증 불필요
```

**B. "질문형" 글쓰기**

```markdown
# React 상태 업데이트가 비동기인 이유는 뭘까?

문서를 보면 "배칭 최적화" 때문이라는데,
정확히 어떤 메커니즘인지 궁금해서 파봤다.

내가 이해한 것:
[...]

아직 모르는 것:
- 내부 큐는 어떻게 관리되나?
- 우선순위는 어떻게 결정되나?

혹시 아시는 분 있으면 댓글 부탁드립니다!
```

모름을 솔직하게 말하는 게 오히려 호감도를 높인다. 그리고 댓글로 배울 수 있다.

**C. "시리즈물" 전략**

```markdown
# Next.js 성능 개선기 #1 - 문제 발견
현재: LCP 4.2초
목표: 2.5초
시도할 것: [...]

# Next.js 성능 개선기 #2 - 실험 결과
시도: [...]
결과: 3.1초로 개선
예상보다 적었던 이유: [...]
다음 시도: [...]
```

"진행형" 시리즈는 완벽할 필요가 없다. 실패도 콘텐츠가 된다.

## 최종 개선안

이론은 충분히 살펴봤다. 
이제 실제로 할 수 있는 것을 정리해보자.

### 즉시 실행: 오늘부터 할 것

**1. 폴더 구조 정리**

```
/블로그
  /주차장
  /작성중
  /Outdated
```

초벌 메모는 따로 쌓지 않는다. 바로 ready로 가거나, 주차장에 한 줄만 남긴다.

**2. 첫 글 형태 결정: "경험 기록" 스타일**

```markdown
# [프로젝트]에서 [작업] 하면서 배운 것

⚠️ 제 프로젝트 경험을 기록한 것입니다

문제: [...]
시도: [...]
해결: [...]
배운 점: [...]

(작성 시간: 15-20분 목표)
```

**3. 15분 타이머 글쓰기 습관**

- 매일 아침 커피 타임에 15분
- 타이머 울리면 문장 중간이어도 멈춤
- 다음날 이어서

**4. 발행 기준 재정의**

```
기존: "완벽하고 검증된 글"
새로운 기준: "15분 안에 쓴 경험 기록"

완성도 30%여도 발행
나중에 업데이트 가능
```

### 중기 전략: 한 달 안에 습관화할 것

**1. 매주 일요일: 주차장 정리**

- 5개 넘으면 우선순위 매기기
- 흥미 없어진 것 과감히 삭제
- 7일 지난 ready 파일은 발행 or 삭제

**2. 월간 회고 글**

```markdown
# 2025년 10월 개발 회고

이번 달 배운 것 3가지:
1. [...]
2. [...]
3. [...]

다음 달 목표:
[...]
```

회고는 검증 부담이 적고, 경험의 집합이라 쓰기 쉽다.

**3. 피드백 루프 구축**

- 발행 후 실제 반응 확인
- 뇌의 부정적 편향 재조정
- "생각보다 괜찮네!" 경험 쌓기

### 장기 목표: 3개월 후 되고 싶은 모습

- 완벽하지 않아도 발행하는 것에 익숙해지기
- "경험 기록" 20개 이상 쌓기
- 그 중 일부를 묶어서 "시리즈" 형태로 발전시키기
- 글쓰기가 부담이 아닌 일상이 되기

_[타임라인 제안: 즉시 → 1달 → 3개월 로드맵을 시각화한 간단한 다이어그램]_

## 마무리: 완벽한 글보다 30%짜리 글 10개

이 모든 탐구의 결론은 간단하다.

**뇌는 완벽을 추구하도록 설계되지 않았다.** 전전두엽의 용량은 제한적이고, 도파민은 시간에 따라 감소하며, DMN은 주제를 확산시킨다. 완벽주의는 편도체가 만들어낸 환상이다.

그렇다면 뇌의 한계를 인정하고, 그 안에서 작동하는 시스템을 만들어야 한다.

완벽한 글 1개를 쓰려다 포기하는 것보다, 30% 완성도의 글 10개를 쓰는 게 낫다. 그 10개가 쌓이면 패턴이 보이고, 글쓰기 신경 회로가 강화되고, 어느새 "잘 쓰는 사람"이 되어 있을 것이다.

오늘 이 글도 완벽하지 않다. 뇌과학 설명이 부정확할 수도 있고, 개선안이 모든 사람에게 맞지 않을 수도 있다. 하지만 괜찮다. 이건 내 탐구 과정의 기록이니까.

**이제 시작이다.** 이 글을 쓰는 데 걸린 시간은... 음, 15분은 아니지만, 그래도 발행 버튼을 누를 것이다.

그리고 내일은 주차장에 있는 "React useEffect cleanup" 주제를 15분 동안 써볼 것이다.
